# MVP Estimate: сроки, команда, бюджет и риски

## 1. Цель оценки

Оценить реалистичный объём работ для первого коммерчески проверяемого релиза сервиса мониторинга сайтов 1С-Битрикс.

Оценка дана для продукта с тремя направлениями:

- SaaS-кабинет;
- Bitrix-модуль;
- базовая подготовка к self-hosted, без полноценного enterprise-набора в MVP.

## 2. Рекомендуемый объём MVP

В MVP включить:

- регистрация и вход пользователей;
- организации и базовые роли;
- добавление сайтов;
- Bitrix-модуль с настройками;
- handshake, heartbeat, metrics batch;
- локальная очередь в модуле;
- collectors: environment, disk, backup, modules, agents;
- внешний uptime из 2 probe-нод;
- SSL check;
- базовый domain check;
- alert engine;
- инциденты;
- Telegram, email, webhook;
- простой dashboard;
- список сайтов;
- карточка сайта;
- список и карточка инцидентов;
- тарифные лимиты без полноценной онлайн-оплаты;
- Docker Compose для dev/self-hosted preview;
- документация установки.

Не включать:

- SMS/voice;
- white-label;
- SSO;
- Kubernetes;
- сложные browser checks;
- полноценный биллинг с рекуррентными платежами;
- deep log analytics;
- мобильное приложение.

## 3. Команда

### Минимальная команда

Подходит для прототипа и закрытой beta.

| Роль | Загрузка |
| --- | --- |
| Product/Project manager | 0.3-0.5 FTE |
| Bitrix-разработчик | 1 FTE |
| Backend-разработчик | 1 FTE |
| Frontend-разработчик | 0.5-1 FTE |
| DevOps | 0.2-0.4 FTE |
| QA | 0.2-0.5 FTE |

Срок: 4-6 месяцев.

Риск: один разработчик часто блокирует несколько направлений, часть качества придётся добирать после beta.

| Frontend-разработчик | 1 FTE |

## 4. Декомпозиция по этапам

### Этап 0. Discovery и технические пробы

Срок: 2-3 недели.

Работы:

- проверить API Bitrix для backup, updates, agents, license;
- собрать spike-модуль без полноценной упаковки;
- проверить HMAC module-to-api;
- проверить внешний uptime из двух локаций;
- выбрать backend-стек;
- уточнить MVP-проверки;
- описать JSON schemas payload-ов;
- подготовить UX wireframes.

Результат:

- подтверждена реализуемость collectors;
- выбран стек;
- утверждён scope MVP;
- выявлены ограничения legacy Bitrix.

### Этап 1. Каркас продукта

Срок: 4-6 недель.

Работы backend:

- auth;
- organizations;
- sites;
- API keys;
- ingest endpoints;
- PostgreSQL schema;
- базовая очередь;
- metrics storage;
- health checks.

Работы модуля:

- install/uninstall;
- options page;
- queue table;
- heartbeat;
- sender;
- HMAC;
- test connection.

Работы frontend:

- login;
- dashboard shell;
- add site flow;
- site list;
- basic settings.

Результат:

- можно добавить сайт;
- модуль подключается;
- heartbeat виден в кабинете.

### Этап 2. Проверки и инциденты

Срок: 5-7 недель.

Работы:

- collectors: environment, disk, backup, modules, agents;
- remote config;
- external HTTP probes;
- SSL checks;
- domain checks;
- alert engine;
- incident lifecycle;
- карточка сайта;
- карточка инцидента;
- графики базовых метрик.

Результат:

- сервис создаёт реальные инциденты;
- пользователи видят технические риски;
- есть история проверок.

### Этап 3. Уведомления и beta-ready UX

Срок: 3-5 недель.

Работы:

- Telegram;
- email;
- webhook;
- notification delivery log;
- mute;
- maintenance windows;
- роли;
- проекты/клиенты для интеграторов;
- тарифные лимиты;
- polishing UX;
- документация установки.

Результат:

- продукт готов к закрытой beta на 5-10 клиентах.

### Этап 4. Stabilization и marketplace preparation

Срок: 3-5 недель.

Работы:

- тестирование на разных Bitrix-окружениях;
- оптимизация тяжёлых collectors;
- безопасность API;
- обработка сетевых сбоев;
- marketplace packaging;
- скриншоты и описание;
- документация передачи данных;
- внутренний admin минимум;
- observability SaaS.

Результат:

- MVP готов к публичному запуску или публикации в Marketplace.

## 5. Сроки

Ориентиры:

| Сценарий | Срок до закрытой beta | Срок до публичного MVP |
| --- | --- | --- |
| Минимальная команда | 4 месяца | 5-6 месяцев |
| Оптимальная команда | 3 месяца | 4-5 месяцев |
| Ускоренная команда | 2-2.5 месяца | 3-4 месяца |

Self-hosted preview можно подготовить к концу MVP, но полноценный enterprise self-hosted лучше планировать отдельным этапом после SaaS beta.

## 6. Бюджетные диапазоны

Оценка зависит от ставок команды. Ниже порядок величин при российском рынке разработки.

### Экономный MVP

- Команда: 2-3 ключевых разработчика + частичный PM/QA/DevOps.
- Срок: 5-6 месяцев.
- Бюджет: 3.5-6 млн ₽.

Что получится:

- рабочий SaaS;
- рабочий модуль;
- базовый кабинет;
- beta для ограниченного числа клиентов.

Риск:

- меньше UX-polish;
- больше ручной поддержки;
- слабее self-hosted и биллинг.

### Сбалансированный MVP

- Команда: 5-7 человек с частичной загрузкой ролей.
- Срок: 4-5 месяцев.
- Бюджет: 7-12 млн ₽.

Что получится:

- рыночный MVP;
- стабильный модуль;
- нормальный кабинет;
- интеграторские сценарии;
- подготовка к Marketplace;
- базовая observability.

### Агрессивный запуск

- Команда: 8-10 человек.
- Срок: 3-4 месяца.
- Бюджет: 12-20+ млн ₽.

Что получится:

- быстрее выход на рынок;
- больше параллельной разработки;
- выше шанс успеть SMS/voice или расширенные отчёты.

Риск:

- без сильного техлида и аналитики параллельность даст технический долг.

## 7. Инфраструктурные расходы MVP

Минимально в месяц:

- VPS/API: 5 000-20 000 ₽;
- PostgreSQL/Timescale: 5 000-30 000 ₽;
- Redis: 2 000-10 000 ₽;
- 2-3 probe-ноды в разных локациях: 3 000-15 000 ₽;
- email provider: 0-5 000 ₽ на старте;
- object storage: 500-3 000 ₽;
- мониторинг самого SaaS: 0-10 000 ₽.

Итого для beta: примерно 15 000-80 000 ₽/мес без SMS/voice.

SMS/voice считать отдельно, так как себестоимость зависит от объёма и провайдера.

## 8. Ключевые риски

### Bitrix API и legacy

Риск: разные версии Bitrix по-разному отдают данные о бэкапах, обновлениях, лицензии и агентах.

Снижение:

- начинать со spike на 3-5 реальных проектах;
- возвращать `unknown`, а не ломать collector;
- документировать ограничения;
- делать collectors отключаемыми.

### Нагрузка на клиентский сайт

Риск: сканирование директорий, таблиц и логов может тормозить сайт.

Снижение:

- лимиты времени;
- кеширование тяжёлых результатов;
- разные интервалы collectors;
- запрет глубокого сканирования в MVP;
- рекомендация cron-режима агентов.

### Ложные алерты

Риск: пользователь отключит уведомления из-за шума.

Снижение:

- debounce;
- подтверждение внешнего uptime двумя probe-нодами;
- maintenance windows;
- mute;
- аккуратные дефолтные пороги.

### Доверие к SaaS

Риск: клиенты не захотят передавать технические данные.

Снижение:

- страница состава передаваемых данных;
- открытый код клиентского модуля;
- self-hosted вариант;
- отсутствие персональных данных по умолчанию;
- понятная политика безопасности.

### Marketplace

Риск: модуль не пройдёт модерацию или публикация затянется.

Снижение:

- готовить модуль по требованиям Marketplace заранее;
- не завязывать beta только на Marketplace;
- поддержать ручную установку через архив.

## 9. Критический путь

Наиболее важные зависимости:

1. Подтвердить collectors на реальных Bitrix-проектах.
2. Сделать стабильный HMAC ingest.
3. Реализовать локальную очередь модуля.
4. Сделать alert engine без дублей.
5. Запустить внешние probe-ноды.
6. Довести Telegram/email уведомления до надёжного состояния.
7. Проверить нагрузку модуля.

Если один из пунктов 1-5 не готов, beta будет выглядеть как демо, а не как сервис мониторинга.

## 10. План закрытой beta

Подобрать 5-10 сайтов:

- 2 свежих Bitrix-проекта;
- 2 legacy-проекта;
- 1 интернет-магазин;
- 1 коробочный Битрикс24;
- 1 проект на shared-hosting;
- 1 проект с большим `/upload/`;
- 1 проект с агентами на cron;
- 1 проект с агентами на hit-режиме.

Метрики beta:

- время установки;
- количество ошибок collectors;
- количество ложных инцидентов;
- среднее время доставки уведомления;
- нагрузка на сайт;
- понятность интерфейса;
- какие проверки пользователи считают ценными.

Минимальная длительность beta: 3-4 недели.

## 11. Критерии перехода к публичному MVP

Можно запускаться публично, если:

- 80% beta-сайтов стабильно присылают heartbeat;
- нет критичных ошибок установки/удаления модуля;
- ложные critical-инциденты ниже 2-5%;
- Telegram/email delivery работает стабильно;
- модуль не создаёт заметной нагрузки;
- есть документация установки и troubleshooting;
- поддержка понимает типовые ошибки;
- есть план тарификации;
- есть публичное описание состава передаваемых данных.

## 12. Рекомендуемый порядок работ

1. Утвердить MVP scope и тарифы.
2. Сделать spike collectors на реальных Bitrix-проектах.
3. Зафиксировать API JSON schemas.
4. Разработать модульный каркас и HMAC ingest.
5. Поднять backend core.
6. Добавить probe-ноды.
7. Реализовать alert engine.
8. Подключить уведомления.
9. Собрать кабинет.
10. Провести beta.
11. Подготовить Marketplace и публичный запуск.

## 13. Что можно отложить без вреда

- SMS/voice.
- PDF-отчёты.
- White-label.
- SSO.
- Kubernetes.
- Deep logs.
- Browser scenarios.
- Полный self-hosted licensing.
- Автоматические remediation scripts.

## 14. Итоговая рекомендация

Оптимальный путь — делать SaaS-first MVP за 4-5 месяцев, параллельно сохраняя архитектурную совместимость с self-hosted. Module-only режим стоит реализовать как ограниченный fallback внутри Bitrix-модуля, но не делать его главным продуктом.

Главная продуктовая ставка: не конкурировать с обычными uptime-сервисами, а дать Bitrix-интеграторам и владельцам сайтов единый контроль технического здоровья проекта.
