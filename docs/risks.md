# Risk Register

## Высокие Риски

| Риск | Влияние | Вероятность | Снижение |
| --- | --- | --- | --- |
| Разные версии Bitrix по-разному отдают данные collectors | Высокое | Высокая | Spike на 3-5 реальных проектах, состояние `unknown`, отключаемые collectors |
| Тяжёлые collectors создают нагрузку на сайт | Высокое | Средняя | Таймауты, лимиты глубины, кеширование, cron-режим, разные интервалы |
| Ложные critical-инциденты вызывают отключение уведомлений | Высокое | Средняя | Debounce, подтверждение двумя probe-нодами, maintenance windows, mute |
| Клиенты не доверяют SaaS технические данные | Высокое | Средняя | Страница передаваемых данных, self-hosted, отсутствие персональных данных по умолчанию |
| Marketplace-модерация затягивается | Среднее | Средняя | Поддержать ручную установку архивом, готовить требования заранее |

## Средние Риски

| Риск | Влияние | Вероятность | Снижение |
| --- | --- | --- | --- |
| WHOIS/RDAP для доменов работает нестабильно | Среднее | Средняя | Делать domain expiry best-effort, не блокировать MVP |
| Webhook URL может стать SSRF-вектором | Высокое | Низкая | Строгая проверка схем, IP, портов и повторный DNS resolve |
| Очереди растут при недоступности downstream | Среднее | Средняя | Backpressure, retry policy, DLQ/failed статус, внутренние алерты |
| Уведомления Telegram/email могут иметь provider failures | Среднее | Средняя | Retry, delivery log, rate limits, fallback-диагностика |
| Self-hosted отвлекает от SaaS MVP | Среднее | Средняя | Делать только Docker Compose preview без enterprise-функций |

## Операционные Метрики Рисков

- Доля collector failures по типу collector.
- P95 времени выполнения collector внутри модуля.
- Количество false-positive critical incidents.
- Notification delivery latency и failure rate.
- Queue size и возраст старейшей pending-задачи.
- Доля сайтов без heartbeat дольше порога.

## Правила Управления Рисками

- Любой high-risk пункт должен иметь owner и проверку в discovery.
- Риск переводится в закрытый только после проверки на реальном окружении или end-to-end сценарии.
- Если риск влияет на безопасность или приватность, решение фиксируется в `docs/security/`.
