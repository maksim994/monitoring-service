# PRD: сервис мониторинга сайтов 1С-Битрикс

## 1. Назначение продукта

Сервис должен помогать владельцам сайтов и Bitrix-интеграторам заранее видеть технические риски: недоступность сайта, истекающий SSL или домен, переполненный диск, отсутствие резервных копий, устаревшие модули, зависшие агенты, проблемы cron и другие типичные отказы Bitrix-проектов.

Продукт не должен быть только внешним uptime-мониторингом. Его ценность — объединить внешние проверки с внутренней диагностикой Bitrix через устанавливаемый модуль.

## 2. Цели MVP

MVP должен подтвердить, что пользователи готовы подключать сайты, получать точные инциденты и использовать сервис как рабочий инструмент сопровождения.

Обязательные цели:

- Подключение первого сайта занимает не более 15 минут.
- Пользователь видит единый статус сайта: доступность, SSL, диск, бэкапы, обновления, агенты.
- Сервис создаёт инциденты только после подтверждения проблемы, без лишнего шума.
- Telegram и email уведомления доходят при критичных событиях.
- Интегратор может видеть несколько сайтов в одном кабинете.
- Модуль не создаёт заметной нагрузки на Bitrix-сайт.

## 3. Не цели MVP

В первый релиз не входят:

- полноценный white-label для агентств;
- SMS и голосовые звонки как обязательные каналы;
- сложные browser-сценарии оформления заказа;
- SSO/SAML/OIDC;
- Kubernetes-поставка self-hosted;
- автоматическое исправление проблем на сайте клиента;
- хранение персональных данных пользователей сайта;
- глубокий APM и профилирование PHP-кода.

## 4. Целевые аудитории

### Bitrix-интегратор

Сопровождает много клиентских сайтов. Ему нужен обзор по всем проектам, группировка по клиентам, SLA-отчёты и быстрый способ понять, где риск.

Боли:

- клиент узнаёт о проблеме раньше интегратора;
- сложно следить за SSL, доменами и бэкапами на десятках сайтов;
- нет единого отчёта о качестве поддержки;
- проверки разбросаны между хостингом, UptimeRobot, Bitrix-админкой и ручными регламентами.

Ценность:

- один кабинет для всех сайтов;
- отчёты для клиента;
- ранние предупреждения;
- эскалации и история реакции.

### Владелец сайта

У него 1-5 сайтов. Нужны простые уведомления и понятная карточка состояния без DevOps-терминологии.

Боли:

- сайт может лежать часами без уведомлений;
- домен или SSL могут истечь внезапно;
- непонятно, работают ли резервные копии;
- обновления и безопасность контролируются нерегулярно.

Ценность:

- простой статус «всё хорошо / есть риск»;
- понятные рекомендации;
- Telegram/email уведомления;
- минимум настроек.

### Enterprise / закрытый контур

Клиент не может отправлять технические данные во внешний SaaS.

Боли:

- требования безопасности запрещают облачные сервисы;
- нужна история и аудит внутри собственной инфраструктуры;
- необходимы роли, audit log и предсказуемая поставка.

Ценность:

- self-hosted версия;
- контроль данных;
- лицензирование по числу сайтов;
- возможность внутренних probe-нод.

## 5. Роли и права

| Роль | Права |
| --- | --- |
| Owner | Управление организацией, оплатой, пользователями, сайтами, ключами |
| Admin | Управление сайтами, проверками, уведомлениями, инцидентами |
| Integrator | Управление сайтами назначенных клиентов и проектов |
| Operator | Просмотр статусов, подтверждение и комментирование инцидентов |
| Viewer | Только просмотр дашбордов, отчётов и истории |
| Self-hosted admin | Установка и обслуживание серверной части в контуре клиента |

RBAC должен применяться на уровне организации, проекта и сайта.

## 6. Основные пользовательские сценарии

### Регистрация и первый сайт

1. Пользователь регистрируется.
2. Создаёт организацию.
3. Нажимает «Добавить сайт».
4. Указывает домен и тип: сайт Bitrix или коробочный Bitrix24.
5. Сервис генерирует `site_id` и `api_secret`.
6. Пользователь устанавливает Bitrix-модуль.
7. В админке модуля вводит API URL, `site_id`, `api_secret`.
8. Нажимает «Проверить подключение».
9. В кабинете сайт получает статус «подключён».
10. Сервис автоматически включает базовые проверки.

Критерий успеха: первый heartbeat и первый batch метрик получены в течение 2 минут после настройки модуля.

### Обработка инцидента

1. Probe-ноды фиксируют недоступность сайта.
2. Alert engine подтверждает проблему по правилу.
3. Создаётся critical-инцидент.
4. Уведомление отправляется в Telegram/email.
5. Оператор открывает карточку инцидента.
6. Нажимает «Принять в работу».
7. Сервис продолжает проверять восстановление.
8. После нормализации инцидент закрывается автоматически.

Критерий успеха: пользователь видит причину, время начала, затронутую проверку, доказательства и историю уведомлений.

### Контроль регламентных рисков

1. Модуль отправляет дату последнего бэкапа, свободное место и состояние агентов.
2. Сервис сравнивает значения с порогами.
3. Создаёт warning-инцидент, если бэкап старше N дней или диск меньше X%.
4. Пользователь получает рекомендацию.

Критерий успеха: владелец сайта понимает действие: освободить диск, настроить бэкап, перевести агентов на cron.

### Работа интегратора

1. Интегратор создаёт клиентов и проекты.
2. Добавляет сайты в проекты.
3. Видит общий список проблем.
4. Фильтрует critical-инциденты.
5. Раз в месяц формирует отчёт по SLA и техническим рискам.

Критерий успеха: интегратор может использовать отчёт как артефакт поддержки клиента.

### Self-hosted установка

1. Клиент получает пакет Docker Compose.
2. Администратор заполняет `.env`.
3. Запускает сервисы.
4. Создаёт первую организацию.
5. Подключает Bitrix-модуль к внутреннему API.
6. Настраивает внутренние probe-ноды.

Критерий успеха: данные не покидают инфраструктуру клиента.

## 7. MVP-функции

### Сайты

- Добавление сайта по домену.
- Автоматический статус подключения.
- Отображение версии модуля.
- Отображение последнего heartbeat.
- Ротация API-ключа.
- Отключение сайта без удаления истории.

### Проверки

MVP-проверки:

- внешний uptime главной страницы;
- внешний uptime произвольного URL;
- SSL expiry;
- домен expiry, если доступно через WHOIS/RDAP;
- свободное место на диске;
- дата последнего резервного копирования;
- версии модулей и доступные обновления;
- состояние агентов;
- версия PHP;
- heartbeat модуля.

### Инциденты

- Создание инцидента при нарушении правила.
- Автоматическое закрытие при восстановлении.
- Статусы: open, acknowledged, resolved, muted.
- Severity: info, warning, critical.
- Комментарии.
- История уведомлений.
- Mute проверки.
- Maintenance window.

### Уведомления

MVP-каналы:

- Telegram;
- email;
- webhook.

Требования:

- тестовое уведомление;
- настройка каналов на сайт и организацию;
- разные каналы для warning и critical;
- повтор critical-уведомлений по интервалу;
- лог всех попыток отправки.

### Кабинет

MVP-разделы:

- общий дашборд;
- список сайтов;
- карточка сайта;
- список инцидентов;
- карточка инцидента;
- настройки проверок;
- каналы уведомлений;
- пользователи и роли;
- тариф и лимиты.

## 8. UX кабинета

### Общий дашборд

Должен показывать:

- количество сайтов всего;
- количество сайтов в critical/warning/ok;
- открытые инциденты;
- сайты без heartbeat;
- ближайшие SSL/domain истечения;
- сайты без свежего бэкапа;
- средний uptime за период.

Основная задача экрана — ответить на вопрос: «Где сейчас нужно вмешаться?»

### Список сайтов

Колонки:

- статус;
- домен;
- проект/клиент;
- последний heartbeat;
- uptime за 24 часа;
- SSL до;
- диск свободен;
- последний бэкап;
- открытые инциденты;
- версия модуля.

Фильтры:

- статус;
- клиент;
- проект;
- severity;
- нет heartbeat;
- SSL истекает;
- бэкап просрочен.

### Карточка сайта

Вкладки:

- обзор;
- проверки;
- метрики;
- инциденты;
- уведомления;
- модуль;
- настройки.

На обзоре должны быть карточки:

- доступность;
- SSL;
- домен;
- диск;
- бэкапы;
- обновления;
- агенты;
- PHP/окружение.

### Карточка инцидента

Должна показывать:

- severity;
- статус;
- время открытия;
- длительность;
- affected check;
- доказательства;
- график значения;
- историю уведомлений;
- комментарии;
- кнопки acknowledge, mute, close manually.

### Настройки проверок

Пользователь должен видеть предустановленные правила и менять:

- включена ли проверка;
- частота;
- пороги;
- debounce;
- repeat interval;
- каналы уведомлений;
- maintenance windows.

## 9. Тарифная модель

### Free

Для первого знакомства.

- 1 сайт.
- Uptime каждые 15 минут.
- SSL, диск, бэкап, heartbeat.
- Telegram и email.
- История 7 дней.
- Без проектов и клиентов.

### Basic

Для владельца нескольких сайтов.

- До 5 сайтов.
- Uptime каждые 5 минут.
- Все MVP-проверки.
- История 90 дней.
- Webhooks.
- До 3 пользователей.

### Agency

Для интеграторов.

- 25+ сайтов.
- Клиенты и проекты.
- Uptime 1-3 минуты.
- Отчёты SLA.
- Роли.
- Расширенная история.
- Шаблоны правил.
- Приоритетная поддержка.

### Enterprise

Для закрытого контура.

- Self-hosted.
- Неограниченные пользователи по договору.
- Audit log.
- SSO на расширенном этапе.
- Персональные условия хранения данных.
- Расширенная поддержка.

## 10. Метрики успеха

Продуктовые:

- activation rate: доля аккаунтов, подключивших первый сайт;
- time to first heartbeat;
- доля сайтов с включёнными уведомлениями;
- количество подтверждённых инцидентов на сайт;
- retention через 30 дней;
- конверсия Free → Basic/Agency.

Технические:

- ложные critical-инциденты менее 2% от всех critical;
- доставка Telegram/email менее 60 секунд после открытия инцидента;
- P95 открытия дашборда до 2 секунд на 100 сайтах;
- ingest API P95 до 300 мс без учёта фоновой обработки;
- потери metric events — 0 при кратковременной недоступности downstream-сервисов.

## 11. Требования к данным и приватности

По умолчанию сервис собирает только технические данные:

- домен;
- версия Bitrix и PHP;
- список модулей и версии;
- состояние проверок;
- технические метрики;
- события инцидентов;
- настройки уведомлений.

Не собирать без отдельного согласия:

- персональные данные пользователей сайта;
- содержимое заказов;
- содержимое форм;
- дампы БД;
- файлы из `/upload/`;
- cookie и сессии пользователей.

В кабинете должна быть страница «Передаваемые данные» с расшифровкой payload-ов.

## 12. Критерии готовности PRD для разработки

PRD считается реализованным на уровне MVP, если:

- описанные роли заведены в RBAC;
- основные сценарии можно пройти в интерфейсе;
- MVP-проверки доступны и имеют дефолтные правила;
- пользователь получает уведомления по инцидентам;
- интегратор может добавить несколько сайтов;
- ограничения тарифов применяются в backend;
- документация объясняет установку модуля и состав передаваемых данных.
